EXPECT: A User-Centered Framework for Developing KBSs

EXPECT’s approach to COA critiquing: a reusable method for plan evaluation
Approach: Design and reuse of a PSM that captures general principles, in terms of
Plan ontology: represents manually or automatically created plans, tasks, goals, commitments, causal links etc
Critiquing and evaluation ontology: “a critique is the result of analyzing a feature of the plan and determining whether it deviates from a given norm”
Evaluation PSMs: e.g., “a goal statement that is missing any of its subcomponents is incomplete”
Current scope of PSM:
Analysis of the plan structure: (e.g., “check missing task decomposition links”)
Analysis of resource use (e.g, “check that non-shareable resources are not used by different tasks that overlap in time”)

Example: Defining a COA critique
Developer defines force ratio critique as a local,           quantity-based critique
Generic problem solving knowledge
For each task,
check that the amount available to the task
is at least as great as the amount needed by the task
Domain-dependent methods added:
Estimate the amount of force-ratio available to a task
Estimate the amount of force-ratio needed by a task
The PSM handles the iteration, comparison and book-keeping required to make the critique.

Natural-language explanation generation
EXPECT’s representation language was developed with explanation generation in mind
Architecture is based on EES [Swartout and Moore 93; Swartout et al. 91]
Thesis: internal representation must reflect how people reason and explain
makes it possible to generate natural language explanations without hand-coded templates
Natural language descriptions of methods are automatically combined into a set of linked hypertext pages to create a critique explanation.
A prototype used for the evaluation achieved a high score for justification soundness.

Template-less NL generation: example
(capability (evaluate (obj (?o is (inst-of task)))
                            (with-respect-to (?f is (spec-of amount-used)))
                            (of (?r is (inst-of resource)))))
(method (check-if
          (obj (estimate (obj amount) (of ?r) (needed-by ?o)))))
          (is-greater-than (estimate (obj amount) (of ?r) (available-to ?o)))))
->
In order to evaluate a task with respect to the amount used of a resource,
   estimate amount of the resource needed by the task,
    then estimate amount of the resource available to the task
    and check if the first amount is greater than the second amount.

Summary
The Expect critiquer both produced high-quality results and demonstrated several key HPKB technologies:
PSM-based critiquer design using middle-level ontologies and generic problem-solving knowledge.
Use of a broad spectrum of HPKB ontologies designed by a number of participants.
Generation of natural-language explanations without hand-coded templates.
Integration of a critiquer with Expect’s knowledge maintenance and knowledge acquisition capabilities (see our KA CCE presentation tomorrow).

Benefits and challenges of template-less NL generation
Benefits:
No additional step for programmer
No opportunity to miss vital steps of the explanation because there is no template
Using the same interface for knowledge acquisition provides a seamless interface for the expert                        (more in KA CCE presentation)
Challenges:
Knowledge engineer must write explicable code        (perhaps this is a benefit?)
The system must be told the “correct” level of detail
